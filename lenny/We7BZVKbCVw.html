当「写代码」被模型接管之后：Claude Code 负责人的下一幕

2026-02-19 · 1h27m · Lenny's Podcast 深度精读
AI 编程 Agent 与工具使用 产品方法论 组织与生产力
Boris Cherny
▶ 观看原视频

01. 高层摘要

  • 嘉宾认为对他所从事的那类编程而言，「写代码」已基本被解决；行业重心正转向模型主动提方案、跨编码的通用任务与更长时的自主运行。
  • Claude Code 从终端原型起步，核心赌注是「为六个月后的模型而设计」——早期产品体验会不舒服，但模型跃迁后产品会突然「对上节奏」。
  • Anthropic 内部数据指向极端的工程杠杆：团队规模约数倍扩张的同时，人均产出（以 PR 等衡量）约增长 200%，与大型科技公司经年累月几个百分点的效率提升形成对比。
  • 产品哲学强调「潜在需求」：观察用户如何「误用」产品完成真实目标，以及观察模型「想做什么」——少把模型塞进固定工作流，多给工具与目标。
  • 组织层面：略「欠编制」会倒逼自动化；对个体工程师应早期慷慨给足 token 做实验，规模化后再做成本与模型档位优化。
  • 角色边界在模糊：PM/设计/工程重叠度或达一半，「软件工程师」称谓可能被「builder」等替代，但跨学科通才与「想清楚做什么」仍被看重。
  • 安全被框定为三层：训练对齐与可解释性、实验室评测、以及真实环境行为；早期外发是为在「野外」学习并与用户反馈共同迭代。
  • 对个人建议：大胆试用最强模型与多形态界面（终端/桌面/移动）、计划模式再执行、并行多 Agent；同时警惕用旧模型时代的习惯指挥新模型。

02. 本期主题与嘉宾背景

嘉宾：Boris Cherny

公司：Anthropic

角色：Claude Code 负责人（Head of Claude Code）

背景：曾在大厂负责多产品代码质量与工程效率；著有 TypeScript 相关技术书；创办过大型 TypeScript 社区；曾在日本乡村生活后加入 Anthropic。播客中自述曾短暂加入 Cursor 后约两周因更认同 Anthropic 的安全使命而回归。

本期主题：Claude Code 一周年回顾、AI 编程 Agent 的产品与安全路径、工程组织与生产力、非技术人群的 Agent 化（Co-work）、对职业与教育的影响。

为什么重要：Claude Code 被视为 Anthropic 增长关键驱动力之一；公开讨论中涉及市场份额类数据（如第三方称 GitHub 提交占比）、内部工程实践与对外产品原则，对理解「编码之后」的行业拐点有一手语境。

适合团队：AI 产品、开发者工具、平台工程、安全与对齐研究、PM/设计、技术管理者（CTO）、增长与运营自动化相关团队。

03. 核心观点精读

观点 1：编码自动化之后，战场移到「提需求、排优先级与长时运行」

核心观点

在嘉宾个人工作流中，代码已由 Claude 全量生成且数月未手改一行，但仍会审阅代码与安全；下一阶段是系统从反馈渠道、缺陷与遥测中主动提出修复与交付想法，并扩展到非编码任务。

关键论据

播客中描述将 Slack 反馈线程交给产品、Co-work 处理日常事务（如罚单、项目协调）、以及模型从「只会聊」走向「会用工具在真实环境行动」。

典型案例

内部长期存在的 Claude Code 反馈频道：早期人工极快响应每一条反馈以建立信任；现阶段由 Claude 先扫频道并提交若干 PR 供人确认。

对公司的启发

应把「需求洞察—优先级—PR 洪流—审查」整条链路的瓶颈拆开投资；对交易所类业务可映射为：工单/客诉/风控事件是否能被 Agent 先聚类、草稿修复或生成变更说明，再人工把关。

观点 2：终端不是怀旧，而是为「模型迭代速度」保留形态弹性

核心观点

最初因单人开发便利选用终端，后来团队判断在模型能力快速爬坡阶段，过重的前端形态可能跟不上模型，因此在一段时间内坚持终端为主战场，再逐步扩展到桌面、网页、IDE 插件、Slack、GitHub、移动端等。

关键论据

嘉宾回忆夜间反复思考「模型持续变好时应如何承接」，并明确将「最小脚手架暴露模型」作为设计取向。

典型案例

ClaudeCLI 早期演示：模型在仅有批处理工具的情况下自行组合能力回答「当前在听什么音乐」类问题，显示工具赋权后的涌现用法。

对公司的启发

大前端团队可评估：是否应以「薄壳 + 强 Agent」迭代，而非每次模型升级都重做复杂向导式 UI；多端一致 Agent 可降低教育成本。

观点 3：「潜在需求」：看用户如何歪用产品，也看模型想怎么走

核心观点

经典 latent demand 是用户用非预期路径完成高价值事（如 Facebook 群组里的买卖、个人主页浏览中的约会信号）；对 AI 产品则多一维——观察模型在分布上的倾向，用最少编排把能力放出来。

关键论据

大量用户用 Claude Code 做种番茄、基因组、恢复婚礼照片、读 MRI 等非编程事，促成 Co-work；产品表述为「产品即模型」，与用户侧「把产品带到人所在工作流」并列。

典型案例

数据分析师学会开终端跑 Claude Code 做 SQL 分析，随后团队内快速扩散——典型「高摩擦入口也拦不住真实需求」。

对公司的启发

埋点与定性研究应显式追踪「非设计用途」与「高摩擦使用」；对金融与合规场景，可把「歪用」转化为受控的专用工作流而非压制。

观点 4：严格编排的 Agent 工作流可能是短期补丁，长期输给更通用的模型

核心观点

嘉宾批评把模型锁进「第一步第二步」的厚重编排器；更优路径通常是给工具与目标，让模型自行取上下文与排序；Richard Sutton「苦涩的教训」被引为心智模型——更通用的能力往往胜过专用堆叠，且专用增益常被下一代模型抹平。

关键论据

指出一年前可能需要大量脚手架，如今许多可删减；脚手架或许带来 10%–20% 提升，但下一版模型到来时可能不值回维护成本。

典型案例

Claude Code 选择押注未来模型而非当下可用性，直到 Opus 4 / Sonnet 4 一类模型出现后出现增长拐点。

对公司的启发

内部平台组应避免过早固化「唯一官方 Agent 流水线」；保留插件化工具与评测，以便模型升级时主要换「大脑」而非重写编排。

观点 5：生产力数据极端化，但「人仍要站在当下模型」

核心观点

Anthropic 在引入 Claude Code 后工程人头与人均产出同时大幅上升；但个人层面需不断刷新习惯，否则会被更「AGI 向前」的新人超越。

关键论据

内存泄漏案例：负责人用手动 heap 分析时，新同事直接让 Claude 抓快照、写临时分析工具并更快提交修复。

典型案例

嘉宾自称每天合并大量 PR，且公司层面 Claude 参与 100% PR 审查，再叠加人工审查。

对公司的启发

培训应包含「何时交给 Agent」的决策模板与复盘；绩效体系需奖励问题定义与验证，而非手写行数。

观点 6：略欠人手 + 极强个体 = 更会用 AI 交付

核心观点

在只投一名工程师的项目上，内在动力会推动其用 Claude 自动化大量工作；与「多给人手再谈效率」相反，适度资源约束可加速工具吃透。

关键论据

嘉宾明确将「略 underfund」与「能今天就做完就别拖到明天」并列为团队原则，并与早期仅靠速度取胜的生存策略相连。

典型案例

Co-work 团队 10 天用 Claude Code 搭出面向大众的产品形态（含虚拟机/护栏等安全考量），并选择较早发布以换真实反馈。

对公司的启发

编制与项目立项可实验「小团队 + 无限 token 预算（初期）」组合；警惕大团队并行却未统一 Agent 规范导致重复劳动。

观点 7：Token 策略：先宽松试验，再规模化优化

核心观点

对 CTO 级建议是先让工程师充分消耗 token 试「疯狂想法」，验证价值后再谈降本（如切换更小模型）；单实验成本相对薪资通常不高，真规模化再优化。

关键论据

公司内部已出现单月 token 费用达数十万美金量级的工程师个案；亦提到「无限 token」正成为一些公司的招聘卖点。

典型案例

对比 Meta 时期全公司代码质量工作：长年累月换来几个百分点的人均提升，与当下三位数百分比量级的变化对照。

对公司的启发

财务与工程需共设「探索预算」与「生产预算」两套规则；合约/交易类高实时系统可在非关键路径上大方试最强模型。

观点 8：安全与发布节奏：实验室好不等于野外好

核心观点

安全分对齐/可解释、合成评测、真实世界行为三层；模型越强，第三层越关键。Claude Code 曾内部试用数月再外发；Co-work 以研究预览姿态早发以收集真实行为数据。

关键论据

提及神经元级监测、欺骗相关概念在模型中的表征、以及开源沙箱让 Agent 在边界内运行；强调「race to the top」与行业共治叙事。

典型案例

Co-work 内置虚拟机与护栏，由 Claude Code 生成实现细节，团队把重心放在「如何更安全、更易引导非工程师」。

对公司的启发

金融与交易所场景必须把「野外层」风控（权限、审计、回滚、人类批准点）产品化，而非仅依赖模型厂商的安全声明。

04. 方法论提炼

Framework核心问题使用场景关键步骤参考价值
潜在需求（Latent demand）双维框架用户正在用什么笨拙方式完成高价值事？模型在不受限时倾向调用哪些能力？发现新产品形态、扩展 Agent 从单点工具到平台、优先级排序。1）从行为数据与社群反馈中寻找「非设计用途」与高摩擦使用；2）将其抽象为可复制的工作流；3）并行观察模型在工具开放时的稳定行为；4）用最小产品包一层安全与引导；5）早发、快收反馈。减少「拍脑袋路线图」；让需求证据与模型演化方向对齐，适合高速迭代期的 AI 产品。
「为六个月后的模型而设计」赌注若最强模型在工具使用、长程任务、计算机操作与 unattended 运行上继续走强，今天的架构会不会一票否决？初创与内部创新项目在早期 PMF 不明显时的路线选择。1）列出确信会随模型变强的能力维度（工具、长时运行等）；2）避免深度绑定当下弱点的编排；3）用可替换的工具层与评测夹住风险；4）接受前数月体验「不对」；5）在模型跃迁周做集中验收与推广。降低每次模型发布后的重写幅度，把竞争力建立在「最先接住新模型」而非「最会包装旧模型」。
安全与对齐三层观测模型在训练与实验室里可靠，是否在真实工具链与权限下仍然可靠？对外发布 Agent、在内网接 Slack/Gmail/交易系统、以及合规审计。1）训练与对齐、可解释性监测；2）合成场景评测；3）受控外发与野外 telemetry；4）把学习反馈回训练与产品护栏；5）开源或标准化沙箱边界以降低生态风险。把「安全」从宣言拆成可运营的发布节奏，特别适合高权限、高金额场景。

05. 对公司的启发

  • 合约与交易前端可评估「同一交易 Agent 内核 + 多端薄壳」：移动端行情、下单与风控确认仍做人机协同，把复杂编排交给模型与工具链，避免每次模型升级重做一套向导。
  • 客诉、风控工单与异常订单调查典型是多工具、长链路任务，可借鉴「计划模式 → 自动执行 → 人审 PR/人审结论」的双闸门，平衡速度与责任认定。
  • 工程效率上可设「探索期 token 预算」：对策略回测脚本、运营工具、内部仪表盘等非核心链路优先最强模型，验证收益后再做模型降级与缓存。
  • Agent 定义应对齐播客中的技术含义——能调用内部 API、工单系统、数据仓库与消息通道，而非仅聊天；交易所场景下工具设计（只读行情、模拟下单、受权真实下单）比提示词工程更关键。
  • 组织上「略欠编制 + 强个体 + 全栈协作」可能加速 AI 采用；公司 可试点小队的 PM/设计/工程共用同一 Agent 规范与评审流，减少交接与等待。
  • 安全与合规视角：参考三层模型，把链上/法币/用户资产相关权限做成硬边界沙箱，野外行为日志与回放作为审计资产，而非事后辩解材料。

06. 公司级行动清单

行动项适用团队适用场景预期收益优先级
建立「反馈—快速可见响应」闭环：公开或内部渠道明确 SLA，鼓励用户/同事提交可复现描述，并用 Agent 先草稿修复或分类。产品工程联合开发者工具、内部平台、用户社区提高有效反馈量与方向准确度，减少「黑箱感」。P0
制定「计划模式 → 执行 → 双审查（自动+人）」的标准工作流，并对高风险变更强制人审。平台工程 / 安全AI 生成代码进入主分支或触发生产配置吞吐提升同时保留问责与回滚点。P0
审计现有 Agent 编排：删除可被更强模型替代的刚性步骤，改为工具清单 + 目标 + 评测。架构 / ML 平台内部 Copilot、运维机器人、数据流水线降低模型换代时的重写成本，提升上限。P1
为工程师设定 3–6 个月的「实验 token」政策与账单看板，规模化后再做模型档位与缓存优化。工程效率 / 财务 BP采纳 AI 编码与 Agent 的早期与中期避免过早省小钱导致探索不足；避免无度扩张导致不可控成本。P1
显式追踪「非设计用途」使用信号，季度复盘是否应产品化为独立模块（类似 Co-work 路径）。产品 / 数据分析用户用交易或行情工具完成意外但高频的任务从未预料需求中发现增长点与风险点。P2
培训新人「先问最强模型能否端到端解决」并配套 heap/日志类问题的标准 Agent 剧本，减少路径依赖旧习惯。工程负责人 / Mentor线上缺陷与性能调查缩短定位时间，拉齐代际工作方式。P1
对权限型 Agent 落地沙箱、操作录像与批准点，并把野外异常纳入安全响应流程。安全 / 合规可访问账户、资金或敏感数据的自动化满足审计与事故追溯，降低「模型失控」尾部风险。P0

07. 讨论问题

1. 当编码产出接近全自动化时，组织如何重新定义「高级工程师」——审阅、系统设计、还是业务与风险的综合判断？

2. 「略欠编制」与员工福祉、单点风险之间如何取舍；哪些项目适合小团队突击，哪些必须冗余人力？

3. 在高度监管行业，早期宽松 token 与快速外发如何与合规、数据出境、模型供应商条款对齐？

4. 设计师与工程师对 AI 工具满意度差异（播客中引用的非正式调查）背后是否是工具形态、评价标准或权力结构问题？

5. 通用模型持续吃掉专用工作流时，公司内部自研模型/微调/小模型的边界应划在哪里？

09. 一句话总结
