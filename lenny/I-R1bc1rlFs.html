用 20 个 AI Agent 替换销售团队之后：Jason Lemkin 的 GTM 实验与 2026 前瞻

01. 高层摘要

  • Jason Lemkin（SaaStr 创始人、资深 B2B 投资人）在末位销售离职后做出激进选择：用 AI Agent 体系替代传统 go-to-market 编制，形成「约 1.2 名真人 + 20 个 Agent」运转、原先约 10 名 SDR/AE 规模的工作被重构。
  • 节目核心论题之一是销售职能正在被 AI 根本性改写：嘉宾侧观点包含「多数 SDR/BDR 在一年内趋近消失」等强判断，并穿插投资组合里各公司采纳 AI GTM 的观察。
  • 实操层面覆盖：具体 Agent 分工、哪些战术有效、部署中的坑与复盘，以及如何把 AI 嵌入销售动作而不丢掉仍有效的人类环节（例如邮件质量从「可接受」到「真的好」）。
  • 对人的启示转向「hyper-employable」：在 AI 普及的销售未来里，何种能力组合更可能进入「能 thrive 的 20%」，以及为何领导岗位相对更不易被简单替代。
  • 战略建议包含「不要轻易自研销售用 AI 工具」的警示、人机分工中「人类监督」的定位，以及 forward-deployed engineer 等角色如何进入叙事。
  • 反直觉点：AI 未必净减少工作量，可能制造更多协调与迭代；同时嘉宾将当下形容为「magical times」，并提出「隐身模式测试」等发现自动化机会的方法。
  • 展望部分涉及 2026 SaaS 与 GTM 走向的预测，以及 AI 对就业结构的广泛影响；节目亦对比人类仍在销售中胜出的场景。

02. 本期主题与嘉宾背景

嘉宾：Jason Lemkin

公司：SaaStr

角色：创始人

背景：全球最大软件创业者社区 SaaStr 的创办者；资深 SaaS/B2B 投资人，曾向 B2B 初创企业部署超 2 亿美元资金；长期以销售组织、年度大会与内容影响硅谷 ToB 圈。

本期主题：以一手实验讨论 AI 如何重塑销售与 GTM：Agent 编排、编制与绩效、工具选型与自建边界、邮件与触达质量、领导与一线角色演变、2026 行业预测与就业影响。

为什么重要：兼具「卖软件的人如何卖软件」与「投遍生态的样本外推」双重视角；其组织实验对正评估 AI 销售编制的创始人、RevOps 与销售负责人有直接对照价值。

适合团队：销售与 SDR/AE 管理、RevOps、市场与增长、客户成功、产品与工程（尤其 forward-deployed）、高管战略与招聘

03. 核心观点精读

观点 1：从「满编销售」到「极少真人 + 多 Agent」的可运行模型

核心观点

在人员流失触发下，GTM 可被重排为少数人类监督 + 大量专用 Agent，承接原有多岗位流水线。

关键论据

节目引用数据对比：曾约 10 名全职销售相关人员；现为约 1.2 名人类与 20 个 Agent（见节目简介与片头 transcript 片段）。

典型案例

SaaStr 自身作为实验场：从实验走向「新运营模型」，用于讨论可推广性与边界。

对公司的启发

评估 AI 销售转型时，应用「编制表 + 任务拆解」显式列出哪些节点由 Agent 闭环、哪些必须人审，避免只换工具不换流程。

观点 2：AI 对销售职能的冲击：从岗位标签到流水线再造

核心观点

变化不仅是效率工具层，而是对触达、筛选、跟进与协作方式的底层重写。

关键论据

官方提纲将「AI 如何根本改变销售职能」列为首位讨论主题，并设专章谈 GTM 在 AI 时代的演变。

典型案例

可对照节目时间轴中 07:13、14:18 等段落对「销售团队如何被重新定义」的展开。

对公司的启发

RevOps 应主导「流程数字孪生」：把漏斗每一步的输入输出、SLA 与质量门槛写清，再映射到 Agent 能力边界。

观点 3：SDR/BDR 的生存危机叙事与 AE 生态位重排

核心观点

节目提出强预测取向：大量 SDR/BDR 角色可能在极短时间内被替代或压缩；AE 与更上游角色需重新定位价值。

关键论据

简介明确列出「为何多数 SDR/BDR 将在一年内『灭绝』」及「SDR、BDR、AE 的未来」章节。

典型案例

结合嘉宾投资组合中不同采纳速度的 GTM 团队，讨论哪些行业/客单价/销售复杂度仍延缓自动化。

对公司的启发

若客单价与关系复杂度低、名单可规模化触达，应优先实验 Agent 化拓客；高信任、长周期大单仍应保留资深关系与方案能力。

观点 4：领导力与「顶层岗位」为何仍相对安全

核心观点

在 AI 渗透销售组织时，承担判断、资源分配、文化与风险兜底的角色更难被完全自动化。

关键论据

章节标题直接设问「为何领导岗位更安全」，并与「如何成为 AI 销售未来里能 thrive 的 20%」相邻讨论。

典型案例

对照 SaaStr 实验中「人类监督」专章（约 53:50）对 oversight 设计与职责划分。

对公司的启发

管理者应从「管人头」转向「管系统与例外」：定义红线、批准策略、处理争议客户与品牌风险。

观点 5：自建 vs 采购：为何节目倾向不要「自己造销售 AI」

核心观点

在工具爆炸期，自研成本与迭代速度可能不如押注成熟栈与组合集成。

关键论据

时间轴含专段「为何不应自建 AI 工具」，并与后文「具体 AI Agent 与应用」衔接。

典型案例

Show notes 引用 Delphi、Vercel、ElevenLabs、Replit 等生态与案例链接，暗示「站在巨人栈上组装」的路径。

对公司的启发

默认策略：买/租核心能力 + 用内部数据与工作流封装差异化；仅当 Agent 与 CRM/合规深度耦合且构成壁垒时再评估自研。

观点 6：Agent 落地痛点：从能跑到真的好

核心观点

部署挑战不仅在技术连通，还在质量曲线与持续学习；邮件与文案类输出要从「可接受」提升到「真的好」。

关键论据

章节「AI 部署中的挑战与学习」与「让 AI 生成邮件变好（而非勉强可用）」形成前后呼应。

典型案例

节目对比「机器可读」与「人类可信」的文本标准，适用于冷触达与跟进序列调优。

对公司的启发

为每类 Agent 输出建立人类评分 rubric（相关性、事实性、语气、合规），并做 A/B 与抽检闭环。

观点 7：人类仍赢的场景：关系、语境与高风险判断

核心观点

销售并未全栈机器化；在特定互动与决策上人类仍有优势。

关键论据

提纲单列「人类在销售中仍击败 AI 的时机」，与 Agent 绩效章节形成张力。

典型案例

可用于大单谈判、复杂多线程采购、品牌敏感事件等「非结构化」情境的护栏设计。

对公司的启发

画一张二维表：任务结构化程度 × 关系/声誉风险；仅低风险高结构区全自动化。

观点 8：「更多工作」悖论与「魔法时刻」心态

核心观点

AI 可能增加协调、提示与迭代负担，但同期能力跃迁也打开新增长面。

关键论据

章节「为何 AI 在制造更多工作而非更少」与「为何称之为 magical times」并列，提示组织预算与期望管理。

典型案例

对创业团队：在 headcount 不增前提下，需接受「运维 Agent 栈」本身成为新职能。

对公司的启发

为 AI 项目单列运营成本（提示工程、监控、数据清洗、合规审查），避免只算模型单价。

观点 9：「隐身模式测试」与机会发现

核心观点

用一种思维实验定位可被自动化的工作流盲区。

关键论据

时间轴 01:25:25 专讲 incognito mode test for finding AI opportunities（节目标题表述）。

典型案例

团队可匿名走一遍内部或外部流程，记录摩擦点与重复劳动，再映射到 Agent 候选。

对公司的启发

与「forward-deployed engineers」段落联动：让工程贴近一线交付以识别真实自动化面。

04. 方法论提炼

Framework核心问题使用场景关键步骤参考价值
人机编制重构五问（基于节目归纳）哪些销售步骤应被 Agent 吞掉，哪些必须保留人？从传统 SDR/AE 编制转向 Agent 编排时的首轮盘点。1）列出端到端漏斗任务。2）标注结构化程度与错误成本。3）匹配现成工具或定制工作流。4）设人审节点与 KPI。5）每季度复盘是否应「升权/降权」自动化。避免一刀切裁人 or 堆砌工具，形成可审计的分工表。
AI 邮件质量阶梯产出从「能发」到「对方愿回」差在哪？冷触达、会议跟进、活动后 nurture。1）定义好邮件的可验证标准（个性化信号、事实准确性、CTA）。2）人机共创样本集。3）线上监测回复率与投诉。4）把失败样本回灌为约束。把生成式 AI 从成本中心转为可度量渠道。
隐身模式走查（Incognito test）若抛开内部政治与历史路径，从零观察流程，摩擦在哪？发现隐藏自动化机会、破除「我们一直这么做」。1）选一条核心客户旅程。2）以新用户/新员工视角逐步记录。3）标重复、等待与手工拷贝。4）评估 Agent 替代与风险。5）优先高收益低风险的批次上线。把机会发现从口号变为可重复方法。

05. 对公司的启发

  • 加密与金融产品的销售链路高度合规与声誉敏感：可自动化部分应限定在高结构、低声誉风险环节（如线索分级、资料寄送、日程协调），把复杂解释与争议处置留在资深人力。
  • 若存在大量零售或机构客户经理，需参照「领导与监督岗更安全」逻辑，提前投资「例外处理」与「关系升级」 playbook，避免 Agent 触达引发监管或舆情事件。
  • 工具策略上警惕自研陷阱：与节目一致，优先 API 与成熟栈拼接，把差异化放在数据、风控规则与客户洞察，而非重复造「通用销售 Bot」。
  • 「AI 带来更多工作」对运营团队同样成立：要为提示、监控、抽检与模型更新预留 headcount 或外包，否则自动化会退化为准自动化烂尾。
  • Forward-deployed 思维可映射到「工程/产品驻场交易或客服」：贴近一线才能发现真实可自动化面，而非会议室臆测。
  • 就业与培训：一线销售与运营需转向 rubric 设计、数据标注、客户研究与高难谈判等「hyper-employable」技能栈，避免仅承担可脚本化劳动。

行动项适用团队适用场景预期收益优先级

07. 讨论问题

1. 我们业务里哪一段漏斗「结构化高、声誉风险低」，最适合作为第一个全 Agent 闭环试点？

2. 若 SDR 职能被压缩，线索质量与品牌语气由谁负责？人审节点设在何处最省成本又不踩线？

3. 自研销售 Bot 与采购栈的边界是什么：我们的护城河究竟在数据/风控还是工作流本身？

4. 当 AI 让「触达」变便宜时，我们如何防止对客户过度骚扰，避免短期指标上升、长期品牌受损？

5. 领导层是否已把「监督 Agent」写进管理与考核，而不仅是传统人头管理？
