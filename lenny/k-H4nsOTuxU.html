01. 高层摘要

  • Anthropic 在约 14 个月内从约 10 亿美元 ARR 量级冲到约 190 亿美元（播客录制时数字已滞后），增长团队坦言主因是研究与模型能力，增长是「全公司战役」而非单团队揽功。
  • 增长负责人 Amol Avasare 将约 70% 时间花在「成功灾难」上：业务极好导致获客、激活、变现各环节被规模击穿，图表全绿但一线情绪压力很大。
  • AI 产品的核心难题之一是「能力悬垂」：模型能力指数上升，而产品侧把能力传导给用户的 onboarding、实验与学习速度跟不上，旧实验结论常被新模型作废。
  • 激活上反复验证的一点：并非一味减摩擦；「对的摩擦」——通过问答理解用户身份与意图、再推荐正确产品/功能——往往提升转化与全链路价值。
  • AI 优先公司的增长组合应更偏向「大赌注」而非仅堆叠小微优化：未来两年产品价值可能是今天的百倍千倍，过度抠小 win 会看不清森林。
  • 内部倡议 CASH（Claude Accelerates Sustainable Hypergrowth）尝试用 Claude 自动化增长实验闭环（发现机会→实现→测试质量→分析学习），目前在文案与小 UI 上已能产生收益，水平约等于 2–3 年经验 PM，跨职能对齐仍高度依赖人。
  • 工程因 Claude Code 等工具杠杆最大，默认「5 工程师 + 1 设计 + 1 PM」在产出上接近管理更多工程师；PM 与设计被挤压，小项目（≤约两周工程）由工程师兼任「迷你 PM」（对接安全、法务等）。
  • 个人叙事上，嘉宾分享创业失败与严重脑外伤康复经历，强调长期游戏、约束带来专注，以及冥想与情绪管理在高压下的作用。

02. 本期主题与嘉宾背景

嘉宾：Amol Avasare（播客与公开资料常见拼写；字幕中有 Amole/Amal 等语音识别变体）

公司：Anthropic

角色：Head of Growth（增长负责人）

背景：曾创办公司（心理健康量化方向，三年后关停），后在 MasterClass、Mercury 负责增长；以冷邮件联系时任 CPO Mike Kreuger 入职 Anthropic，当时并无公开的 growth GM 招聘。

本期主题：超高速收入增长下的增长职能、组织形态、激活与自动化增长实验、AI 时代 PM/设计/工程协作。

为什么重要：提供头部 AI 应用公司在「模型驱动增长」与「增长团队可操盘空间」上的一手叙事，并指向行业趋势：增长工作可被 AI 部分自动化，但品牌、安全与跨团队对齐仍难替代。

适合团队：增长（获客/激活/变现）、增长平台/实验、各产品线增长 pod（如 B2B、Claude Code、知识工作者、API）、产品、设计、工程、数据、法务合规（onboarding 与实验常涉及）

03. 核心观点精读

观点 1：「成功灾难」是超增长期的默认状态

核心观点

当 acquisition、activation、monetization 同时被规模放大时，会出现大量紧急救火；增长负责人需心理上调适「全绿曲线」与「极度疲惫」并存。

关键论据

嘉宾估计约 70% 时间用于 success disasters，仅约 30% 为更传统的主动增长规划。

典型案例

类比 Facebook、Uber、DoorDash 早期极速扩张时的体验。

对公司的启发

把「幸运的问题」写进管理预期：排障与扩容能力要和增长假设同等优先级。

观点 2：能力悬垂让激活与实验「永远追不上模型」

核心观点

模型一代代解锁新能力时，团队刚跑完测试、上线新引导，下一版模型可能已使旧假设失效；行业瓶颈从产品「有没有智能」转向「能不能把智能交到对的用户场景里」。

关键论据

嘉宾描述 Opus 迭代前后能力集变化快于实验迭代周期。

典型案例

若用户习惯仅问天气等浅层 query，再强模型也展示不出差异价值。

对公司的启发

需建立「随模型发布刷新激活地图」的机制，而非一次性 onboarding 项目。

观点 3：「好摩擦」：用分步收集身份与意图来换匹配与转化

核心观点

删除一切步骤并非最优；能解释「产品为何适合你」、并据此路由到正确功能的摩擦，常与更高完成率及长期生命周期价值共存。

关键论据

Anthropic onboarding 被外界认为偏长，但内部数据满意；Mercury 曾整季只做合规 onboarding 体验质量，结果显著；MasterClass、Calm 等测验式购买流被反复验证。

典型案例

多字段表单拆成多屏降低认知负荷，在 Mercury 等处表现更好。

对公司的启发

可沉淀用户画像用于后续生命周期与投放 lookalike，形成复利。

观点 4：AI 优先产品的赌注组合：大 swing 与小微优化再平衡

核心观点

传统增长可能 60–70% 中小实验、20–30% 大赌注；AI 核心价值指数放大时，应显著提高大赌注比重，避免只收割线性改进而错过新市场（如 agentic coding 相对旧 AI 编程市场）。

关键论据

嘉宾对比「杂货配送类应用两年价值 +30–50%」与「AI 产品两年可能两个数量级以上」的叙事；内部更看 log-linear 曲线。

典型案例

增长团队主导 Chrome 扩展以支撑 Co-work、Claude Code 等用例——在另一家公司可能不会做。

对公司的启发

若产品价值非 AI 核心，则不必机械照搬该比例，要看 AI 在价值主张中的位置。

观点 5：增长组织：横向平台 + 受众竖井 pod

核心观点

多产品、多受众时，单一「全产品激活团队」难以服务差异巨大的利益相关方；用 monetization/growth platform 等横向能力，叠加 B2B、开发者、知识工作者等 pod 保持聚焦与对接产品主团队。

关键论据

团队约 40 人，结构接近传统增长团队但受众切分更细。

典型案例

Claude Code 增长与 Kat、Boris 等产品负责人紧密协作。

对公司的启发

结构随阶段调整，核心是 focus 与跨职能 tie-in，而非一劳永逸架构。

观点 6：CASH：用 Claude 闭合增长实验循环

核心观点

对「发现机会、构建、测试品牌与安全底线、分析结果」四环节分别 eval 与爬坡；人类在环审批品牌敏感改动，未来可用品牌 skill 降低审阅成本。

关键论据

Opus 4.5 前效果不足，4.6 后方向更清晰；当前小规模以 copy 与轻 UI 为主，已「印钞」但胜率约 junior PM 水平。

典型案例

嘉宾认为增长领域大量小优化天然更早适合自动化。

对公司的启发

跨职能 stakeholder alignment 被点名仍为人类高地，「六个 Agent 对齐」仍是玩笑式难题。

观点 7：工程杠杆 > PM/设计杠杆时的组织补丁

核心观点

Claude Code 等工具使工程师产出倍增，名义编制不变却相当于管理更多工程师；PM 与设计承压，通过招更多 PM、以及让工程师在短周期项目上承担 PM 职责来缓解。

关键论据

≤约两周工程的项目默认工程师驱动法务安全等沟通，PM 顾问式介入。

典型案例

Claude Code 类产品 PM 本身偏工程背景时匹配度更高。

对公司的启发

招聘「产品嗅觉」工程师与明确 RACI 比单纯加人头更能稳住交付。

观点 8：冷邮件入职：渠道、打开率与坚持跟进

核心观点

高意向职位可个人邮箱 + 验证过的主题行 + 极短正文 + 多次跟进直至对方明确拒绝；本质是增长人自己的漏斗优化。

关键论据

Mike Kreuger 回复冷邮，嘉宾自称其唯一由此招入的 PM。

典型案例

避免只在 LinkedIn/工作邮箱红海触达。

对公司的启发

对关键人才与关键客户，可建立类似「高触达、短文案、多跟进」 playbook。

04. 方法论提炼

Framework核心问题使用场景关键步骤参考价值
增长实验自动化四段循环（CASH 方法论内核）模型在「找事做、做出来、验质量、读数据」四步各有多强，能否周周爬坡？高频、数据驱动、改动面相对可控的增长实验；需品牌与安全护栏。1）机会识别（趋势与历史实验学习）→ 2）构建待上线改动 → 3）质量与品牌 gate → 4）上线后与数据复盘；对每段分别打分与迭代。把低阶实验吞吐交给 AI，人类专注高赌注与跨团队对齐，缩短「能力悬垂」带来的落地延迟。
好摩擦 vs 坏摩擦这一步是否在帮助用户理解「为何选我」并路由到正确体验？注册、KYC、开通交易、开通合约/杠杆、策略跟单等强合规或强路由场景。1）区分无价值耗时与可解释的分步 → 2）用问答/测验收集意图与风险承受 → 3）个性化推荐产品与风控提示 → 4）A/B 验证完成率与长期留存/ARPU。避免「极简漏斗」导致错误用户进错产品路径，降低客诉与资损风险。
指数视角下的赌注组合（Bet Portfolio）两年后的产品价值是线性小幅提升还是数量级跃迁？这如何改变实验组合？AI Native 产品、交易工具、智能投顾/策略类功能等强技术曲线业务。1）画出核心价值是否随模型/数据飞轮指数变化 → 2）若为指数，提高大 swing 资源占比 → 3）保留必要小微优化以复利 → 4）用 log 坐标审视增长，避免被短期噪声带偏。在「大机会窗口」与「当下可交付」之间取得战略平衡，防止过度局部最优。

05. 对公司的启发

  • 合约/现货/跟单/多产品线并行时，可采用「横向增长中台（实验、定价、归因）+ 按客群或产品线分 pod」降低协作扇出，与 Anthropic 多受众结构同理。
  • 交易与资管产品的「激活」同样存在能力悬垂：行情与工具升级快于用户教育；需随大版本发布刷新新手任务、风险教育与策略示例，否则留存实验结论快速过时。
  • KYC 与风险测评不是纯粹损耗：合规前提下分步澄清投资经验、品种偏好、风险承受，可提升产品路由质量（类比 Mercury onboarding 质量季）。
  • 可试点「AI 增长实验助手」：从埋点异常、漏斗跌落、文案与入口微改动起步，人类终审涉及品牌承诺、合规表述与高风险入口的改动。
  • 工程师已普遍用 AI 加速交付时，需显式规则化「谁在小项目上承担 PM」（例如两周规则），避免 PM 成为唯一瓶颈，同时守住法务与风控签字链。

06. 公司级行动清单

行动项适用团队适用场景预期收益优先级
建立「模型/大版本发布 → 激活与新手路径」同步清单，每次大发布强制复盘首会话关键路径。增长 + 产品 + 教研/内容新模型、新交易功能或新跟单能力上线后，旧引导失效或用户仍停留在浅层用法。缩短能力悬垂周期，提高首周激活与核心功能渗透率。P0
审计注册与开通流程：标记「坏摩擦」与「好摩擦」，对意图收集步骤做 A/B 与长期 LTV 联动评估。增长 + 设计 + 合规漏斗极短但用户进错产品线或风险错配，导致投诉与流失。提升匹配度与留存，减少无效获客成本。P0
定义增长自动化试点范围：文案、轻量 UI、内部工具脚本四段循环，并设置品牌/合规 skill 与人类在环。增长平台 + 数据 + 法务实验吞吐不足、排队过长，工程师忙于重复性小改动。提高实验速度，释放 PM 做高赌注项目。P1
对多产品线公司梳理 pod 边界与对口产品负责人，明确每个 pod 的北极星与共享中台 SLA。增长负责人 + 各产品线负责人API、机构、零售、跟单等多轨并行导致优先级冲突。减少跨团队摩擦，加快决策。P1
在研发流程中落地「短周期项目工程师兼任 PM」规则，并模板化法务/风控对接 checklist。工程负责人 + PMO + 合规工程杠杆上升后 PM 成为排队瓶颈。缩短交付周期且不绕过风控。P1
管理层沟通中引入「成功灾难」叙事：为峰值场景预留 on-call、容量与客户成功资源，避免仅庆祝指标。高管 + 运营 + 客服活动或行情带来流量激增、系统与人工支持同时告急。降低团队耗竭与品牌危机概率。P2
用 log-linear 仪表板复盘核心增长指标，区分指数阶段与饱和阶段，校准赌注组合。数据 + 战略线性刻度掩盖早期真实增速或掩盖放缓信号。战略对话与资源分配更一致。P2
关键招聘或关键 BD 目标建立「高触达冷启动」 playbook（渠道、跟进节奏、极简价值陈述）。人才招募 + 商务竞争白热化下传统渠道响应率低。提高顶级人才与战略合作伙伴触达效率。P2

07. 讨论问题

1. 当 AI 自动化实验的胜率接近资深 PM 时，组织如何重新定义 PM 绩效与晋升，而非仅看「 shipped 数量」？

2. 「好摩擦」在强监管金融市场中的边界在哪：教育价值与转化优化何时会与披露义务冲突？

3. 多产品线公司中，横向增长平台与 pod 的权力划分（预算、排期、数据所有权）怎样设计才能避免二次官僚化？

4. 能力悬垂加剧时，是否应把更多资源从 A/B 测试转向定性研究与实时助手式 onboarding？

5. 工程高杠杆背景下，设计职能如何避免沦为「修图外包」，仍参与大赌注的定义与验证？

09. 一句话总结

一句话总结：当模型把价值曲线拉成指数时，增长团队的主要矛盾不再是「找不到增长点」，而是消化成功、用对的摩擦完成激活，并把实验闭环交给 AI——同时承认会议室里的人类对齐仍是瓶颈。
