设计流程已死？Anthropic 设计负责人 Jenny Wen 谈 AI 时代的设计分工、信任与招聘

01. 高层摘要

  • 本期 Lenny’s Podcast 嘉宾 Jenny Wen 曾任 Claude 设计负责人，现主导 Claude Co-work 设计；此前在 Figma 带领 FigJam 与 Slides 团队，并曾在 Dropbox、Square、Shopify 任职。她提出：被业界当作「福音」的经典设计流程——大量调研、发散收敛循环、长期守护流程本身——在工程侧被 AI 极大加速后已基本「失效」；设计被迫跟着变，而非设计圈单方面发起革命。
  • 她将当下设计工作粗分为两类：其一为支撑实现与交付——工程师可用多种工具快速出 scrappy 版本，设计师往往没时间再做「精美高保真稿」，更优策略常是少当拦路虎、让团队先跑起来（let them cook），再以咨询式反馈纠偏；其二为方向与愿景——但技术迭代太快，愿景 horizon 从数年收缩到约 3～6 个月，载体不一定是华丽叙事 Deck，而可能是能指方向的代码原型。
  • 她自述几年前约 60%～70% 时间在画稿与原型，如今mock/原型约 30%～40%，另有 30%～40% 与工程师结对、白板与评审，并有一块时间亲自写代码做最后一英里打磨。Anthropic 日常包含大量「跟上公司内部信息流」：模型研究、各团队代号项目、内测产品试玩——设计角色越来越像帮执行对齐、串联想法、保持体验整体感，而非只交稿。
  • 工具栈上她以 Claude 系为主：对话逐步迁到 Co-work 处理长程任务；Claude Code 多在 VS Code 里改前端，也尝试在 Slack/移动端 @Claude 修小问题再收 PR。Figma 仍不可替代：适合并行探索 8～10 个方向及微调视觉交互；纯跟 coding agent 迭代往往过于线性、易过早锁死单一方向。
  • 质量与信任：早发「研究预览」可以，但要承诺持续迭代、响应反馈；若早发后长期不改进才真正伤品牌——「用速度建立信任」的另一面是让用户感到被听见。关于人类价值：AI 的品味与判断会变强，但仍需有人对「做什么、不做什么」负责；许多最难问题本质是人际分歧而非写代码。
  • 管理与 IC：她入职 Anthropic 先做 IC、短暂带团队后又回到全职 IC，认为在流程剧变期贴近一线能积累硬技能与管理同理心，并类比工程团队让 EM 轮岗摸技术的做法。招聘上她看重韧性与适应力，并归纳三类理想画像：强通才（多块「长板」的 block 型）、极深专才（T 的竖杠特别长）、以及被低估的「高潜应届生」——学习快、包袱少。
  • 她还介绍「可读性框架」（创始人/想法各分可读 vs 难读的四象限）：前沿实验室里设计师可像内部 VC，识别能量高但尚未被讲清的原型（如内测的「Claude Studio」）再转化为叙事或形态——Co-work 中部分信息架构便来自此类探索。领导力方面：她反驳「管理者只做高杠杆事」——资深者亲自测产品、修 bug、为同事用心准备卡片等「看似低杠杆」行为，往往极高杠杆；团队间适度玩笑（roast）若建立在心理安全与高标准并存之上，反而是健康信号。

嘉宾：Jenny Wen

公司：Anthropic（Claude / Claude Co-work）

角色：Claude Co-work 设计负责人；曾任 Claude 设计负责人

背景：前 Figma 设计总监，带队 FigJam 与 Slides；曾在 Dropbox、Square、Shopify 任设计师。活跃于设计前沿与 AI 产品形态探索。

本期主题：AI 崛起下设计流程的重构、设计与工程协作比例、愿景时间尺度、Anthropic 设计师日常、Claude/Co-work/Figma 工具组合、早期发布与信任、人类决策责任、聊天与图形界面未来、IC 与管理、Co-work 产品叙事与内测原型、招聘三类画像、低杠杆工作的再定义、团队心理安全、可读性框架与闪电轮书单产品等。

为什么重要：为「工程已 AI 化、设计如何跟」提供一线头部 AI 公司内部视角：既有对经典流程的告别，也保留 Figma、研究、原型与原则化辅导；对评估自家产品团队组织形态、发布节奏与设计师技能栈有参照价值。

适合团队：产品设计、UX 研究、前端工程、设计系统、增长与品牌、技术文档与开发者体验、设计管理与人事

03. 核心观点精读

观点 1：经典「信任流程」设计方法论在加速交付面前失焦

核心观点

长时间调研—发散收敛—再守护流程本身的做法，在工程师可并行开多条实验分支时已难以成为瓶颈；设计更常被工程节奏倒逼重塑，而非单独宣言「我们要革命」。

关键论据

Jenny 在柏林演讲标题即「不要迷信设计流程」；并指出 Opus 4.6、假期里 Claude Code 普及等近几个月变化让数月前的判断都显得偏旧。

典型案例

工程师先做 scrappy 可试版本，设计师再介入对齐与打磨，而非等完整设计闸门。

对公司的启发

交易所与金融科技产品若工程侧已高频发版，设计流程需显式定义「何时拦、何时跟跑」，避免流程正义拖慢学习闭环。

观点 2：设计工作二分：执行支撑 vs 缩短的愿景

核心观点

一类工作是陪跑实现、咨询式反馈与串联体验；另一类是指向未来的愿景，但 horizon 缩短到约季度级，载体可能是可运行原型而非宏大 Deck。

关键论据

她对比过去 2～5～10 年愿景与当下「不知道两年后世界怎样」；强调在人人能开分支的世界里更需要统一方向以免浪费。

典型案例

Co-work 以研究预览推出，承认粗糙但相信内核价值。

对公司的启发

合规与风控类产品同样可用「预览 + 明确迭代承诺」管理预期，前提是路线图与对外沟通一致。

观点 3：非确定性 AI 逼出「真机真用户」验证

核心观点

模型输出难以用静态稿穷尽状态；可点击原型也不够，需要在真实模型与真实用例下观察，边做边发现场景。

关键论据

「无法 mock 所有状态」；Lenny 补充用户常出乎意料地使用 AI 产品。

典型案例

Co-work 等产品方向部分来自对内对外使用方式的意外发现。

对公司的启发

跟单、客服、行情助手等 AI 功能应以线上灰度与埋点反馈为主证据链，而非仅靠线框验收。

观点 4：时间分配从「大部头视觉」转向「结对与实现」

核心观点

mock/原型占比从约 60%～70% 降到约 30%～40%；与工程师 jam、结对显著上升；并有一块亲自写代码做 polish。

关键论据

嘉宾直接给出前后比例估算并描述与工程师合改首版、自己收尾的体验。

典型案例

IDE 里改 class/CSS 比反复让 agent「把这类改成那类」更快。

对公司的启发

设计系统若以代码与设计 token 为真源，设计师掌握工具链可减少与工程的往返成本。

观点 5：Figma 仍是「多方向探索」的最佳画布

核心观点

好设计需要同时扔多个方向再策展；与 agent 写代码往往线性深钻单一方向。Figma 适合 8～10 方案并行及字体、微交互等细粒度尝试。

关键论据

她自称前 Figmate，仍感到「就该用 Figma」；并对比 coding 工具的线性。

典型案例

重大导航或信息架构变更前在 Figma 批量试错再收敛到实现。

对公司的启发

活动页、交易界面大改版仍可坚持「多方案 Figma → 单一实现分支」，避免 agent 过早锁设计。

观点 6：信任来自速度 + 被听见的闭环

核心观点

早期发布可接受瑕疵，但要公开承诺迭代、快速修问题并回应反馈；早发后停滞才真正侵蚀品牌。

关键论据

研究预览标签、团队推特互动与「昨天已修」的公开节奏被点名。

典型案例

Co-work 标注研究预览并持续迭代首页与任务列表心智。

对公司的启发

上新功能、新市场活动时，对外话术需与真实迭代资源匹配，避免「一次性亮相后无声」。

观点 7：人类剩余价值在问责与组织政治，不止「审美品味」

核心观点

AI 会在品味与判断上变强；但谁为上线负责、两人争论功能边界时谁拍板，仍是人类硬问题。

关键论据

类比工程师仍需为 AI 写出代码是否真可用负责；放射科签字隐喻被提及。

典型案例

Boris 例：Claude Code 已参与想点子与选型，但最终仍要人按确认。

对公司的启发

产品委员会、风险评审与跨部门优先级仍需要明确 owner，而非假设模型输出即策略。

观点 8：界面未来：聊天/终端长期存在 + 可点击控件与生成式 UI 并存

核心观点

Chat 带来对模型的无限灵活接口；但股票、天气等仍用控件更高效；未来更多界面或由模型生成而非逐屏手搓。

关键论据

Claude 内 widget；WhatsApp/Telegram 等「另一种聊天入口」也是创新。

典型案例

Kevin Weil 侧证：对话适配不同智商水平的沟通方式。

对公司的启发

App 内助手可组合「固定关键路径控件 + 自然语言兜底」，降低纯聊天摩擦。

观点 9：「可读性框架」：在难读想法里做内部 VC

核心观点

创始人与想法各有可读/难读；双可读往往不够新；设计师可识别能量高但难讲清的内测原型，再用叙事与 UX 翻译给组织与市场。

关键论据

Evan Tan（SPC）二维框架；Claude Studio 内测界面起初「看不懂」但研究侧热情高，后析出 skills/指令等资产。

典型案例

从复杂 agent harness 界面中抽出计划、待办、上下文文件等 Co-work 有效模块。

对公司的启发

内部 hackathon 与高研原型需设计伙伴早期介入「翻译」，避免只有工程师懂价值。

观点 10：领导力：低杠杆杂活可以是高杠杆信号

核心观点

资深管理者亲自 dogfood、抠 bug、甚至提交 PR、为同事手写卡片，常被教科书归为低杠杆，实则塑造标准与信任。

关键论据

她举高管重度用产品、Mike Krieger 自己提 PR、纪念卡片等例。

典型案例

团队模仿她口头禅「next steps」开玩笑——反映心理安全。

对公司的启发

业务负责人定期走查核心交易路径，比只看汇总报表更能发现体验债务。

观点 11：心理安全与高标准可并存（「严父」模型）

核心观点

同事间能互相打趣不等于无纪律；前提是信任基底厚实，且 leader 明确追求卓越——类似关爱但要求严格的养育方式。

关键论据

她澄清并非强迫 roast，而是舒适区内的玩笑标志安全感；并提醒经理与下属做朋友需边界。

典型案例

crit 上被团队模仿口头禅仍能保持产出质量。

对公司的启发

评审文化可从「只说好话」转向「敢怼细节但不人身攻击」，适合高 stakes 金融产品迭代。

04. 方法论提炼

Framework核心问题使用场景关键步骤参考价值
设计工作二分法（执行支撑 / 短周期愿景）当前阶段主要矛盾是帮团队 ship 得一致，还是为接下来数月统一方向？工程已开多分支时的资源分配、路线图工作坊、设计 headcount 规划。1）盘点并行实验数量与对外承诺 → 2）为执行侧设「不阻塞但要对齐」的触点（crit、设计系统）→ 3）为愿景侧定 3～6 月可证伪假设 → 4）用原型或灰度替代长 Deck → 5）每轮回顾调整两类时间占比。避免设计同时想「控全场」与「造远景」而精疲力竭，又与工程现实对齐。
可读性四象限（创始人 × 想法）这个原型/提案是「难读但能量高」还是「易懂但平庸」？谁能把它翻译为可传播叙事？内测功能、0-1 产品、研究转产品的 handoff。1）在 Slack/内测群中扫信号 → 2）标记高能量难读项 → 3）设计师或 PM 深度访谈开发者 → 4）抽象用户可感知的价值主张 → 5）选最小载体（界面、demo、故事）对外。减少「好东西死在看不懂」的组织损耗。
早期发布与信任契约若现在上线，我们能否展示持续改进而非一次性曝光？研究预览、beta、新市场首登。1）明确已知缺陷与适用人群 → 2）设反馈入口与响应 SLA → 3）公开小步快跑记录 → 4）对内同步修复优先级与资源 → 5）达不到承诺则收紧发布范围。用速度换学习的同时控制品牌与合规预期。

05. 对公司的启发

  • 合约/现货/跟单等多线并行时，工程若已采用 AI 辅助高频改动，设计职能应写入明确的「陪跑节奏」（例如每日 standup 对齐体验回归点），避免只依赖传统里程碑评审。
  • 交易与资产界面强调信任与风控文案：若采用「研究预览」式发布，需与客服、合规及公告渠道同一口径，防止用户误以为功能已完全稳定。
  • 非确定性助手（行情解读、策略说明、客服）应以线上真实会话与埋点为主验收；静态设计稿仅覆盖模板态，需预留异常与模型漂移的 UI/兜底策略。
  • 设计系统与组件库若以代码为真源，可鼓励 UI/UX 掌握 Claude Code 等工具做 token 级修改，缩短与 iOS/Android/Web 工程的最后一公里摩擦。
  • 招聘上可对标三类画像：多技能通才（产品+视觉+实现）、单一维度顶尖专才（如动效/图标/技术美术）、以及高潜校招——在流程剧变期，后者有时比「流程惯性大的资深」更适配。
  • 内部创新管道可借鉴「可读性框架」：量化团队、策略产品、增长实验常有「难读」原型，需设计伙伴早期介入翻译，避免只有少数人理解价值。

07. 讨论问题

1. 当 AI 让「人人能开分支」成为默认，设计系统与品牌规范应更多靠自动化检查还是靠文化共识？

2. 愿景周期缩短到季度级后，战略岗位与设计的衔接是否会变得更像连续实验组合而非年度路线图？

3. 若 AI 品味持续提升，设计教育应更强调叙事、伦理与组织决策，还是仍强化手工视觉基本功？

4. 在受监管金融场景中，「研究预览」式发布与适当性、风险提示义务如何对齐？

5. 心理安全与高标准的平衡在不同文化（全球化团队）中如何落地，而非简单复制 roast 式玩笑？

08. 关键表达与金句
