工程师变「巫师」、脚手架被模型吃掉：OpenAI 平台工程负责人 Sherwin Wu 谈未来 12～24 个月的 AI 与 API

01. 高层摘要

  • 本期 Lenny’s Podcast 嘉宾为 OpenAI API 与开发者平台工程负责人 Sherwin Wu。他给出内部量化信号：约 95% 工程师日常使用 Codex；合并进主干的 PR 100% 经 Codex 评审；重度使用 Codex 的工程师开 PR 数量比轻度用户高约 70% 且差距在拉大。他本人与多位工程管理者的个人代码亦主要由 Codex 产出。
  • 他将软件工程角色变化概括为：一线 IC 越来越像「技术负责人」——同时管理大量 Agent 线程、检查进度、纠偏上下文，而非逐行手写。他援引 MIT《SICP》把编程比作「咒语/巫术」的隐喻，并延伸到当下：自然语言指令即咒语；同时提醒《魔法师的学徒》风险——高杠杆也意味着模型可能失控，资深者仍需把关。
  • OpenAI 内部有一支实验团队维护「100% 由 Codex 编写」的代码库且刻意不设「撸袖子自己写」的退路；由此暴露的典型问题是 Agent 做不出想要的功能。学习之一是：多数失败源于上下文不足或规格 underspec，需要把部落知识写进注释、结构、Markdown、skills 等，让模型可检索。团队计划就经验发博文。
  • Code review 与 CI：Codex 承担大量「无聊但必要」的审查与 lint/修复自动化，把人工评审从「全程盯」降到约 30% 注意力；小 PR 有时可主要信赖 Codex + 作者自检。多模型方面：对外部模型用得少，更强调 dogfood 与内部变体以获得不同视角。
  • 管理侧变化慢于 IC，但趋势是：工具放大顶尖个体产出spread，经理应把更多时间花在头部绩效者身上（ unblock、组织清障）；ChatGPT 接组织知识后，绩效回顾、跨文档研究变容易，未来经理或可管理更大 span。
  • 宏观上他看好「一人十亿美元公司」叙事，但认为市场未充分计价二阶效应：创业与垂直 SMB 软件大爆发、B2B SaaS「黄金时代」、大量千万美元级小公司；VC 追求千倍回报的项目池可能收缩，但对高能动性个体更友好。与 Lenny 讨论支持成本时，他认为软件生产成本下降会带来更多外包给小型定制工具商的组合。
  • 企业 AI 部署：他观察许多落地可能负 ROI，硅谷易活在泡沫里；成功模式常是「高层买单 + 一线兴奋与自下而上最佳实践」双轮驱动，而非纯行政命令。建议组建内部「突击队/tiger team」做深度探索与知识传播，成员常是「工程相邻」的技术爱好者而非必有全职工程师。
  • 产品哲学：在 API/Agent 脚手架领域，「模型会把你的脚手架当早餐吃掉」——向量库、Agent 框架、skills 文件等「时尚脚手架」会随模型变强而贬值；应平衡客户当下诉求与模型 12～24 个月走向。建议「为模型将要去的地方而建设，而非只为今天」。未来 6～18 个月他期待：模型可连贯执行更长时间任务（多小时级）、原生多模态尤其语音/音频在企业场景被低估将爆发。
  • 对创业者：市场极大，失败多因产品未击中客户而非被 OpenAI「碾压」；OpenAI 自视为生态平台公司，新模型一般同步进 API、保持平台中立。API 栈从底层 Responses API（长跑 Agent）到 Agents SDK 再到 Agent Kit/Eval 等更高抽象。结尾寄语：未来 2～3 年仍是技术与创业最有趣阶段之一，建议亲身使用工具而非被资讯洪流淹没。

02. 本期主题与嘉宾背景

嘉宾：Sherwin Wu

公司：OpenAI（API 与开发者平台）

角色：OpenAI API 与开发者平台工程负责人（Head of Platform Engineering）

背景：曾在 Opendoor 负责定价模型等工作；工程与管理经验丰富，现主导服务全球 AI 创业与企业的 API/平台能力。

本期主题：OpenAI 内部 Codex 采用率与 PR 工作流、工程师角色向「Agent 管理者」演化、SICP/魔法师的学徒隐喻、全 Codex 代码库实验与上下文工程、Codex 代码评审与 CI 自动化、工程经理在 AI 时代的 span 与头部绩效者策略、「一人公司」多阶效应与 B2B 机遇、企业 AI 负 ROI 与自上而下反模式、tiger team、客户反馈 vs 模型路线图、脚手架被模型吞噬、Responses API/Agents SDK/Agent Kit、长程任务与音频多模态前瞻、平台使命与创业建议等。

为什么重要：提供来自「AI 经济基础设施」负责人的一手组织数据与平台判断：对评估自家是否低估上下文工程、是否只有行政驱动缺乏自下而上、以及产品应绑定「模型进步曲线」还是「当下局部最优」有直接参照价值。

适合团队：平台与开发者关系、基础设施与 SRE、工程效率与内部工具、企业解决方案与交付、产品与技术文档、技术品牌与社区、组织发展与人事

03. 核心观点精读

观点 1：量化信号：Codex 已成 OpenAI 工程默认工作方式

核心观点

绝大多数工程师每日使用 Codex；所有合并 PR 经 Codex 评审；高用量者产出显著更多 PR，组织能量集中在工具成熟度与信任曲线上。

关键论据

嘉宾给出约 95% 日活、100% PR Codex 审查、70%+ PR 数量差异等内部观察，并引用 Kevin Weil「这是模型最差的一刻」强调预期管理。

典型案例

管理者个人代码亦主要由 Codex 生成，体现工具已覆盖非一线编码场景。

对公司的启发

若贵司仍在「试点 AI 写代码」，可对标本数据设定 adoption、PR 吞吐、评审耗时等北极星，而非仅统计许可证开通数。

观点 2：IC 工程师 →「管理 Agent 舰队」的技术负责人

核心观点

工作从手写转向并行管理多线程任务：检查 Agent 状态、给反馈、防跑偏；乐趣与风险同时上升。

关键论据

描述同事同时开 10～20 个线程（非全时跑满）并频繁 context-switch；与 Lenny 对「Agent 不顺时的焦虑」共鸣。

典型案例

全 Codex 代码库团队无法用传统「自己上手改」兜底，被迫系统化解决 Agent 协作问题。

对公司的启发

绩效与职级标准可能需要纳入「上下文设计、任务拆解、评审治理」而不仅是代码产量。

观点 3：隐喻：SICP 的「咒语」+《魔法师的学徒》的风险

核心观点

编程语言史就是降低对机器「念咒」门槛的历史；自然语言驱动 Agent 是下一阶。高杠杆咒语需要巫师 discipline，否则扫帚失控。

关键论据

回忆《计算机程序的构造和解释》将工程师比作巫师；对比 Mickey 放任扫帚的桥段与 vibe coding 极端情况。

典型案例

与「神灯/猴爪」类比并列：愿望必须精确，副作用需预见。

对公司的启发

内部培训可用统一隐喻对齐风险意识：鼓励用 Agent，但明确「负责人不可休眠」的治理底线。

观点 4：Agent 失败常是上下文与规格问题，不是「模型完全不行」

核心观点

underspec 与部落知识未入库时，Agent 易卡死；解法是把可检索知识结构化进仓库（注释、结构、MD、skills）。

关键论据

100% Codex 实验团队的经验归纳；计划对外分享博文。

典型案例

当功能做不出来时，团队不能退回手写，只能增强文档与规格。

对公司的启发

可审计「仓库对 Agent 友好度」：README、边界情况、示例、skills 是否覆盖关键路径。

观点 5：Codex 评审 + CI 自动化：把人类从无聊环节解放

核心观点

Codex 擅长代码审查；结合自动修 lint/重启 CI，可把评审从十几分钟缩到数分钟；人类注意力从 100% 降到约 30%。

关键论据

嘉宾回忆 Quora 新闻流模块每日几十 review 的痛苦对比；现用 Codex 作「聪明的第二双眼睛」。

典型案例

小 PR 在组织内可更多依赖 Codex + 作者，而非堆人力。

对公司的启发

需定义哪些 PR 类仍强制双人审（安全、合规、核心账务），避免「学徒扫帚」式自我循环盲区。

观点 6：管理：头部绩效者放大效应与「外科医生」支持模型

核心观点

AI 工具放大顶尖个体差距；经理应花 >50% 时间在约 top 10% 身上 unblock。《人月神话》外科手术室隐喻：经理像护士团队托举主刀。

关键论据

让全 Codex 实验团队「放开跑」被点名为高杠杆；组织瓶颈更多在流程与跨部门。

典型案例

Lenny 联想到 a16z「AI 让好人更好、让极强的人极强」——嘉宾认同并加码时间分配建议。

对公司的启发

对交易平台类组织：识别 Agent 高玩并让其沉淀 playbooks，比平均摊派培训更有效。

观点 7：「一人十亿美元公司」的二阶：创业潮与 B2B SaaS 黄金时代

核心观点

市场常讨论一人独角兽，却少讨论伴随而生的百倍小型定制软件商、垂直工具与 VC 回报结构变化。

关键论据

垂直 AI 工具增多；可能出现大量 $10M～$50M 公司对个人已足够改变命运。

典型案例

与 Lenny 论支持工单规模化：未来可能由许多小型播客/通讯专用支持 SaaS 拼接外包。

对公司的启发

生态位上既可做平台，也可做「大创作者/大客户的专用外挂栈」——分发与信任将更关键（Lenny 补充）。

观点 8：企业 AI 负 ROI：硅谷泡沫 + 缺自下而上

核心观点

许多员工只做很基础的提问；纯 CEO 驱动、与真实工作脱节的 AI 倡议易失败；需高层支持与一线狂热并存。

关键论据

观察非科技行业用户的真实熟练度；对比 X 上 power user 的 skills/MCP 文化。

典型案例

OpenAI 内部 Codex 普及由员工自发应用引爆，而非仅靠口号。

对公司的启发

设立 tiger team：常由 Excel 高手型运营/支持负责人等非传统工程师牵头探索与内训。

观点 9：客户声音 vs 模型方向：脚手架会被「吃掉」

核心观点

若盲听客户对向量库/框架的增量需求，可能修在局部最优；模型变强后更简单架构反而胜。Nicholas（Fin 相关实践）语：models eat your scaffolding for breakfast。

关键论据

回顾 2022 以来 agent 框架、向量栈热度的起伏；skills/agents.md 亦可能未来被模型内化。

典型案例

OpenAI API 团队亦承认曾走弯路，随模型进步不断「吃苦教训」。

对公司的启发

路线图需显式字段：「若 12 个月后模型多模态/长程推理提升 X%，我们哪些模块应删除而非加强」。

观点 10：产品建议：为「模型将到之处」而设计

核心观点

做今天 80 分可用、明天模型升级后跃迁到 95 分的产品，而非假设能力静止。

关键论据

以某版本模型「突然解锁」产品体验为例说明拐点思维。

典型案例

创业公司在能力曲线上押注成功者众。

对公司的启发

合约/交易类产品中，可把「模型可承担任务时长」纳入功能分层：从分钟级交互走向可监督的多小时批处理。

观点 11：12～18 个月：更长连贯任务 + 企业音频被低估

核心观点

SWE-bench 类指标上多小时任务成功率在爬升；产品多按 10 分钟级交互设计，未来将不同。音频/语音原生多模态在企业通话、运营中潜力大。

关键论据

引用公开趋势图感受斜率；强调世界大量商务靠语音完成。

典型案例

已有人把 Codex 推到多小时任务，但仍属例外。

对公司的启发

客服、合规录音分析、销售辅助等可提前布局语音链路，而非只卷文本 coding 场景。

观点 12：OpenAI 平台论：市场够大、API 是使命延伸

核心观点

创业公司失败主因是没人爱产品而非被 lab 碾压；OpenAI 视自身为生态平台，模型普遍进 API，强调中立与 foster ecosystem。

关键论据

Cursor 等例；ChatGPT App 生态与 Apps SDK 与 API 团队协作；使命第二条「让益处扩散到全人类」需平台伙伴。

典型案例

Sign in with ChatGPT 等实验亦指向生态。

对公司的启发

可减少对「巨头随时抄你」的过度内耗，把注意力放在分销与不可替代工作流深度。

04. 方法论提炼

Framework核心问题使用场景关键步骤参考价值
双轮驱动 AI 落地（Top-down buy-in × Bottom-up adoption）是否既有高层的工具预算与愿景，又有一线「发烧友」在真实工作流里迭代最佳实践？企业级 Copilot、内部 Agent、跨部门自动化。1）识别各职能中的技术相邻高潜者 → 2）组建专职或半专职 tiger team → 3）按职能场景做 last-mile 实验与文档 → 4）办内部分享与模板复制 → 5）把采纳与成果纳入认可体系而非空 KPI。降低「买了没人会用」的负 ROI 概率。
上下文工程检查清单（Agent 卡死时）失败是因为模型能力，还是规格/可检索知识不足？仓库 onboarding、功能开发、跨服务改动。1）复现最小失败任务 → 2）审计输入上下文是否含边界与反例 → 3）补充 MD/skills/注释/目录约定 → 4）用更小任务验证 → 5）沉淀为可复用提示或规则文件。把「不可写」转为「可协作」，提高 Agent 上限。
脚手架生命周期（Scaffolding vs Model Trajectory）这层编排多少是临时拐杖，多少应长期存在？RAG、工具路由、子 Agent、skills 仓库设计。1）标注每模块依赖的模型能力假设 → 2）设定 12～18 个月能力情景 → 3）为「模型变强」写删减测试 → 4）对客户承诺区分「稳定能力」与「过渡脚手架」 → 5）季度复盘是否应简化。避免在局部最优上过度投资，被模型进步一次性清零。
PR 与 CI 的人机分工（Codex 时代）哪些变更可由 Agent 评审闭环，哪些必须人类强制闸门？高频小改动、内部工具、核心金融路径。1）分类 PR 风险等级 → 2）默认跑 Codex 评审 + 自动修复轻量问题 → 3）定义人类最低审阅清单 → 4）监控「盲信」导致的事故 → 5）迭代信任边界。吞吐上升同时保留安全底线。

05. 对公司的启发

  • 工程团队可对标 OpenAI 内部指标：Codex（或等价工具）日活率、经 AI 辅助的 PR 占比、评审耗时分布；避免仅采购不度量真实工作流嵌入。
  • 合约/跟单/风控等核心路径应明确「Agent 不可单人闭环节点」：与高杠杆工具配套的是上下文规格、强制评审等级与回滚策略。
  • 中文与多语言客服、语音工单、销售录音若占运营比重，可并行试验音频原生模型能力，不过度押注纯文本 Copilot。
  • 企业客户侧推行 AI 时，避免「高管 KPI 式强推」：应找各业务线的 Excel/运营高手型 champion 组成 tiger team，做可复制的场景 playbook。
  • 产品与技术路线需写入「模型进步情景」：长程任务（多小时）若成真，网格/策略回测/批量对账类工作流的产品形态可能从「逐步点选」转向「可监督的长跑任务」。
  • 创业与平台合作心态：细分工作流深度 + 分发能力往往比「害怕 OpenAI 抄功能」更值得投精力；API 能力普遍开放时，护城河在数据闭环与合规交付。
